## Scenario a тАФ Install path

- **Time taken:** ~7 minutes elapsed including two false starts; ~25 s once the right profile was picked.
- **Steps that worked:**
  - `bash deploy/sandbox/scripts/start-sandbox.sh --profile lab` тАФ clones the repo (<1 s via `git clone --local`), generates the layered `.env`, seeds `data/` with the synthetic fixture, brings up the compose project, and waits on `/api/health`.
  - `/api/health` returned `{"status":"ok","db":true,"llm":true}` after 7 probe attempts.
  - Resource isolation confirmed: `docker compose ls` shows `featcat-sandbox-20260514-095841-3583986` distinct from any other project; volumes and networks all carry the project prefix.
- **Steps that broke** (verbatim output kept short):
  - **`--profile local` hung in `bun install` at 0% CPU for ~10 minutes.** Root cause: BuildKit inherits `HTTP_PROXY` from the operator's shell, regardless of the `--env-file` passed to `docker compose`. The local profile already sets `HTTP_PROXY=` in its env file, but that only flows into runtime containers, not the build phase. Fix shipped in commit `0118c9c тАФ fix(sandbox): unset proxy env vars for local profile builds`.
  - **`--profile local` then failed with "DeadlineExceeded: failed to fetch oauth token"** once the proxy was cleared тАФ this host *does* need the proxy to reach `registry-1.docker.io`. Diagnosed as: operator was on the lab profile all along; `--profile local` was the wrong flag for this machine. The launcher's pre-flight already emits `[warn] --profile local but HTTP_PROXY is set in the shell; sandbox will run without proxy` тАФ that warning should arguably escalate to an error when the host shell has a proxy *and* `daemon.json` has a proxy configured. Filed as a UX note below.
  - **`--profile lab` initially ran `compose build` and would have re-attempted the same failed pull-then-build chain.** Fix shipped in commit `568bb1a тАФ fix(sandbox): skip compose build on the lab profile` so lab honours `SANDBOX_REQUIRE_CACHED_IMAGES=true` and skips the build phase entirely.
- **UX confusion:**
  - The `--profile local` vs `--profile lab` choice is the single most surprising decision for a new operator. The launcher warning is easy to miss in a long output stream. Consider auto-detecting (`if HTTP_PROXY set OR /etc/docker/daemon.json has a `proxies` block тЖТ require `--profile lab`).
- **Severity:** `minor` for the missing auto-detect (a single warning already exists); the underlying bugs were fixed during the dry-run and committed.
- **Suggested fix:**
  - Add an explicit error if `--profile local` is picked but the host advertises a proxy (env OR `daemon.json`). Force the operator to retype `--profile local --force` to opt out.

## Scenario b тАФ Doctor / preflight

- **Time taken:** <2 s for the in-image `featcat doctor`.
- **Steps that worked:**
  - `bash exec-sandbox.sh --id <id> -- featcat doctor` succeeded and printed:
    > `All checks passed!` with `Mode: Local`, `SQLite catalog exists`, `LLM server running at http://llm:8080`, `Cache: 0 active entries`.
- **Steps that broke:**
  - `featcat doctor db` (and `doctor llm`, `doctor data`, `doctor network`, `doctor deploy`) all fail with `Got unexpected extra argument (db)`. The shipped `nxank4/featcat:latest` image predates the doctor sub-app split (`refactor(cli): doctor becomes a sub-app routed through the diagnostics runner`, commit `3ffa378`) which landed on `staging` but hasn't been re-published. This is exactly the kind of "developer-tree тЙа shipped image" gap the sandbox is for.
- **UX confusion:**
  - A new operator following the in-repo docs that mention `featcat doctor db` will be confused. Either ship a fresh image to `nxank4/featcat` or document that the sub-app form requires a build.
- **Severity:** `major` for image staleness vs docs (will trip up anyone walking the docs).
- **Suggested fix:**
  - Pin `nxank4/featcat:0.4.x` to the latest staging build, then bump the tag in `deploy/docker-compose.yml`. Mid-term: a CI job that publishes per-merge images so the published `latest` doesn't drift.

## Scenario c тАФ Source registration

- **Time taken:** <1 s.
- **Steps that worked:**
  - `featcat source add demo /sources/synthetic.parquet --description "Sandbox synthetic fixture"` returned `Source added: demo -> /sources/synthetic.parquet`.
  - `featcat source list` rendered the source in a rich table cleanly. Long path was wrapped, not truncated.
- **Steps that broke:** none.
- **UX confusion:**
  - First-time users may not realise `--name` is positional (`source add <name> <path>` not `source add <path> --name`). The error message we hit on the host (`No such option: --name`) before discovering the order was clear enough.
- **Severity:** `nit`.
- **Suggested fix:** add a one-liner usage example to `featcat source add --help` showing the positional order.

## Scenario d тАФ Bulk inventory + browse + search

- **Time taken:** ~2 s total (scan + list + search).
- **Steps that worked:**
  - `featcat source scan demo` registered **7 features** from the synthetic Parquet (matches the schema: `customer_id`, `age_bucket`, `plan_tier`, `arpu_synthetic`, `usage_gb`, `churn_flag`, `signup_month`).
  - `featcat feature list` rendered all 7 with correct dtypes (`int64`, `double`, `large_string`) and null ratio 0.0 % (the synthetic fixture has no nulls by design).
- **Steps that broke:**
  - `featcat feature search "churn"` returned `No features matching 'churn'` despite `demo.churn_flag` being in the catalog. The shipped image's search likely depends on `feature_docs.short_description` being populated; with no docs yet, the search index has nothing to match against. The newer FTS5 work mentioned in CLAUDE.md (tokenises tags, names, column names) appears not to be in the published image.
- **UX confusion:**
  - A first-time user who scans then searches will find this surprising. The CLI gives no hint that docs need to be generated for search to work.
- **Severity:** `major`. Worse than scenario b's gap because the user can't easily work around it from the CLI alone.
- **Suggested fix:** either publish the image with FTS5 across feature names/columns, or have the search command print a `[dim]` hint when zero hits and zero docs exist.

## Reproduce this run

```bash
# 1. Clone the source repo and check out the sandbox branch.
git clone git@github.com:codepawl/featcat.git ~/work/featcat
cd ~/work/featcat
git checkout feat/sandbox-uat-tooling

# 2. Confirm the LLM model is present.
ls -l deploy/models/gemma-4-E2B-it-Q4_K_M.gguf   # ~533 MB

# 3. Pick the right profile.
#    --profile local : your shell has no HTTP_PROXY and Docker can reach the
#                      public registry directly.
#    --profile lab   : you're on a corporate-proxy host (HTTP_PROXY env or
#                      `proxies` block in /etc/docker/daemon.json); cached
#                      images required.

# 4. Start the sandbox.
bash deploy/sandbox/scripts/start-sandbox.sh --profile lab

# 5. Walk through scenarios.
SBOX_ID=<id-from-step-4>
bash deploy/sandbox/scripts/exec-sandbox.sh --id "$SBOX_ID" -- featcat doctor
bash deploy/sandbox/scripts/exec-sandbox.sh --id "$SBOX_ID" -- \
    featcat source add demo /sources/synthetic.parquet --description "demo"
bash deploy/sandbox/scripts/exec-sandbox.sh --id "$SBOX_ID" -- featcat source scan demo
bash deploy/sandbox/scripts/exec-sandbox.sh --id "$SBOX_ID" -- featcat feature list

# 6. Tear down.
bash deploy/sandbox/scripts/reset-sandbox.sh --id "$SBOX_ID"
```
