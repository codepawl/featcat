## Scenario e — Auto-doc generation

### e.1 — single feature

```
$ time featcat doc generate demo.churn_flag
Done: 1 features documented
real  2m2.38s
```

`featcat doc show demo.churn_flag` then printed a well-shaped doc:

> A binary indicator flagging whether a customer has churned within the observation period.
>
> This feature is a binary target variable indicating customer churn (1) or retention (0). It is typically derived from historical customer lifecycle events or a defined observation window. Churn probability is influenced by customer demographics (age_bucket), service value (ARPU), plan tier, and usage patterns (usage_gb). Edge cases include potential mislabeling of historical churn events or data leakage if features are used improperly during model training.
>
> Expected range: Binary (0 or 1)
> Potential issues: Potential issues include class imbalance if churn is rare, data leakage if features are not strictly time-sequenced, and potential mislabeling of the churn status. Ensure the definition of 'churn' aligns with the business definition.

- **Time taken:** 2m02s on lab CPU (no GPU).
- **Worked:** doc rows persisted, `doc show` rendered all 5 fields, model_used logged as `gemma-4-E2B-it-Q4_K_M.gguf`.
- **Broke:** nothing.
- **UX nit:** progress is silent on a non-TTY pipe. The `featcat doc generate <name>` form has no progress indicator; only the `--all` form uses rich.progress. Operators scripting against this command see no output until completion.
- **Severity:** `nit` for the silent progress.

### e.2 / e.3 — combined: `doc generate --all`

The shipped image has **no `--source` filter and no `--limit` flag** (`featcat/cli.py:1533` only accepts `<name>` positional + `--no-cache`, `--all`, `--context`). Collapsing e.2 + e.3 into one `--all` run:

```
$ time featcat doc generate --all
Generating docs... ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 100% 0:00:00
Done: 7 features documented
real  9m38.71s
```

`featcat stats` confirmed `Doc coverage: 100.0% (7/7)`.

- **Time taken:** 9m38s for the full batch on lab CPU. That includes re-generating `demo.churn_flag` (already done in e.1) because the operator used `--all`. The default `featcat doc generate` with no args targets only undocumented features and would have done 6 in roughly 6 × 96 s = 9.6 min — so the per-feature cost is consistent with the e.1 singleton (96 s vs 122 s) once the model is warm.
- **Worked:** progress bar renders cleanly on a TTY (we captured the final `100%` line); per-feature docs persisted; `featcat stats` reflects 7/7 doc coverage; `model_used` populated.
- **Broke:** nothing functional. The checklist's `--source` and `--limit` flags do not exist; closest behaviour is `doc generate --all` (force) or `doc generate` (missing only). See *Documentation drift* table.
- **UX confusion:** the `Done: N features documented` total is the **catalog total**, not the delta from this invocation. After `--all` on a partially-documented catalog there's no easy way to know "how many were regenerated this run." Counter-intuitive when reading the message alone.
- **Severity:** `minor` for the misleading "Done: N" message; `major` for the missing `--source` / `--limit` filters (documented but not shipped).
- **Suggested fix:** rename the message to `Done: N features in catalog have documentation` and add a `--source <name>` filter behind a feature flag if not before. Until then, document the gap in `featcat doc generate --help`.

## Scenario g — Definitions + audit log

```
$ featcat feature set-definition demo.churn_flag --sql "SELECT 1 FROM device_logs WHERE complaint_count > 5"
Definition set: demo.churn_flag (sql)

$ featcat feature info demo.churn_flag
…
  Definition (sql):
    SELECT 1 FROM device_logs WHERE complaint_count > 5
…

$ featcat feature history demo.churn_flag
  v3  2026-05-15 02:49  "definition: None -> 'SELECT 1 …'; definition_type: None -> 'sql'"  unknown
  v2  2026-05-15 02:48  "Generated documentation"                                            unknown
  v1  2026-05-15 02:46  "Initial registration via scan"                                      system
```

### Real findings

1. **`featcat feature diff demo.churn_flag --v1 1 --v2 2` reports "(no differences)"** — even though v1 was the initial registration and v2 was the doc generation, which clearly changed `short_description`. The diff command appears to compare only a subset of fields and miss doc content. **Severity: major.**

2. **Rollback prompts interactively by default; `--yes` is required for scripting.** With `--yes`:

   ```
   $ featcat feature rollback demo.churn_flag --version 1 --yes
   Rolled back. New version 3 created.
   ```

   But `featcat feature history` still shows **only 3 rows** (v3 from the definition-set, v2 from doc-gen, v1 from scan) — **no v4 record of the rollback itself appears.** The "New version 3 created" message either references the current v3 (misleading) or the rollback row was created but the history view caps at 3 rows. **Severity: major** — operators tracking changes through history will not see the rollback event.

### `/api/versions` — checklist URL is wrong

The checklist's `curl http://…/api/versions?feature=demo.churn_flag` returns the SPA index.html (the catch-all swallows the unknown path). The real endpoint is `/api/features/by-name/versions?name=demo.churn_flag`:

```json
[
  {"version": 3, "change_type": "definition", "change_summary": "definition: None -> 'SELECT 1 …'", "changed_at": null},
  {"version": 2, "change_type": "doc",        "change_summary": "Generated documentation",          "changed_at": null},
  {"version": 1, "change_type": "metadata",   "change_summary": "Initial registration via scan",    "changed_at": null}
]
```

`/api/versions/recent` (different path) also exists and returns full snapshot objects per change, including `previous_value` / `new_value` diffs that the CLI `feature diff` is failing to surface.

- **Severity:** the `/api/versions` URL drift in the checklist is a `nit` (docs vs. API). The `changed_at` being `null` in the per-feature response is `minor` — the field exists but isn't populated; the `/api/versions/recent` shape has `created_at` instead, suggesting two slightly different serialisers.

## Scenario h — AI chat SSE

Three POSTs against `/api/ai/chat` with `{"query": "…"}`. SSE stream observed in all three:

### h.1 — "List all features in the demo source"

```
event: message  data: {"type":"tool_start","tool":"list_features"}
event: message  data: {"type":"tool_call","name":"list_features","params":{"source":"demo"}}
event: message  data: {"type":"tool_result","name":"list_features","result":"Showing 7 of 7 matching features:\n- demo.age_bucket (large_string)\n- …\n- demo.usage_gb (double)"}
… token stream …
event: message  data: {"type":"done"}
```

Pass. Tool-call wiring works; 7 features echoed back; clean close.

### h.2 — "Which features could predict customer churn?"

```
event: message  data: {"type":"tool_call","name":"search_features","params":{"query":"predict customer churn"}}
… search result + token stream of clarifying questions …
event: message  data: {"type":"done"}
```

Pass functionally — the agent calls `search_features`, gets nothing semantically relevant (the synthetic fixture's docs are shallow), and **falls back to asking clarifying questions** ("Would you like to filter by source? Suggest based on a use case?"). That's the correct behaviour, not hallucination.

### h.3 — "Tell me about feature foo.bar_nonexistent"

```
event: message  data: {"type":"tool_call","name":"get_feature_detail","params":{"feature_name":"foo.bar_nonexistent"}}
event: message  data: {"type":"tool_result","name":"get_feature_detail","result":"Feature 'foo.bar_nonexistent' not found."}
… token stream …
data: {"type":"token","content":"Feature 'foo."}
data: {"type":"token","content":"bar_nonexistent' not"}
data: {"type":"token","content":" found."}
event: message  data: {"type":"done"}
```

**Pass — no hallucination.** The agent echoed the tool result verbatim instead of inventing a fake feature.

### Findings

- The chat request body schema is `{"query": str, "session_id": str?}` — **not** `{"messages": [...]}` as the checklist implies. Returning a 422 on the OpenAI-style body cost 30 s of trial-and-error. **Severity: minor.** Suggest aligning the SSE endpoint payload with the typical chat-completions shape, or at minimum document `query`-only in the API.
- The agent defaults to **Vietnamese output** ("Đây là thông tin cho bạn:" prefix) regardless of input language. This matches operator locale but may surprise an English-only audience. **Severity: nit.**
- Tool calls are concurrency-safe with background work: I was running `doc generate --all` on the LLM at the same time as h.1; both completed without errors. The model serves one request at a time but llama.cpp's queueing was invisible.

## Scenario i — Similarity graph

```
$ featcat embed --all
sentence-transformers is not installed. Run: uv pip install -e '.'
```

**The shipped `nxank4/featcat:latest` image does not include the optional `[embeddings]` extra.** Embedding-backed similarity is therefore unavailable. The matrix and graph endpoints still respond, but via the TF-IDF fallback path.

### `/api/features/similarity-graph?limit=50`

```json
{"nodes_len":7,"edges_len":0,"sample_node":{"id":"demo.age_bucket", "source":"demo","has_doc":true,…},"sample_edge":null}
```

7 nodes, **0 edges**. The graph view will render an isolated node cloud with no force layout to converge on. Either (a) the TF-IDF threshold filters out every pair, or (b) the graph endpoint requires embeddings to populate edges. **Severity: major** — the checklist's "Graph renders within 5s, force-directed layout stabilizes" expectation cannot hold with the shipped configuration on this fixture.

### `/api/features/similarity-matrix?ids=<7 uuids>`

Requires a comma-separated `ids` query param (per OpenAPI; the checklist's no-arg curl returns 422). Shape:

```json
{"features_len":7, "cells_len":21, "threshold":0.0,
 "cached_at":"2026-05-15T02:53:29.513565+00:00",
 "sample_cells":[{"a":0,"b":1,"score":0.0421},{"a":0,"b":2,"score":0.0421},{"a":0,"b":3,"score":0.0421}]}
```

- 21 cells = C(7,2), correct.
- **Every pair scored 0.0421.** Uniform low score across all 21 pairs strongly suggests TF-IDF is operating purely on feature-name token overlap (`demo.X` vs `demo.Y`) with no per-pair signal once docs are factored in. The matrix is technically populated but conveys no information.
- **Severity:** `major` — the matrix view will render a uniform-coloured grid, defeating the visualisation's purpose.
- **Suggested fix:** make the `[embeddings]` extra part of the standard image (operators pulling `latest` shouldn't have to install extras to use the headline similarity feature), or surface a clearer "this catalog has no embeddings" banner in the matrix UI.

## Top friction (operator-visible)

1. **`feature diff` shows no differences** between any two adjacent versions in this run. Either the diff comparator is matching only metadata fields the test didn't change, or the comparator is buggy. Either way the audit story is broken: an operator looking at history sees rows but cannot see what changed.
2. **Similarity matrix is uniform** without the `[embeddings]` extra. The default `nxank4/featcat:latest` image therefore renders this headline visualisation in a degenerate state. Either ship `[embeddings]` by default or surface a clearer "embeddings not available" affordance in the UI.
3. **No per-source / per-limit filter on `doc generate`.** For a real catalog with hundreds of features the operator has only the all-or-one binary choice, with no way to slice by source or chunk-size.

## What I'd change about `start-sandbox.sh` based on this experience

- Print the path to the seeded fixture in the success summary so the operator can copy-paste `featcat source add demo /sources/synthetic.parquet …` without having to look it up.
- Auto-add the source + scan automatically (behind a `--seed` flag) so the operator skips the duplicate scenarios-c-d steps on every UAT run.
- Reduce the wait-on-health timeout from 90 s to ~30 s for the lab profile (the cached image starts in ≈25 s; 90 s is overkill).

## Production-likeness verdict

Tooling end: production-like. Sandbox starts cleanly, isolates resources, tears down without trace.

Data end: **obviously test data.** The 50-row synthetic fixture's columns (`age_bucket`, `arpu_synthetic`, …) drive the LLM to perfectly plausible but generic descriptions. Doc *generation* exercise is real; doc *quality* assessment needs a richer fixture. **Recommend adding a second fixture** with ≥ 1000 rows and 20+ columns mimicking a real telco source, so the `[embeddings]` and similarity story can be assessed honestly.
